목적

이 문서는 Cursor + Codex가 프론트 작업을 일관된 규칙으로 진행하도록 하기 위한 플레이북이다.

관련 문서

- 프로젝트 개요/요구사항: `frontend-brief.md`
- 스프린트 계획: `dev-plan-frontend.md`

작업 우선순위

- “보이는 화면”이 끊기지 않도록 UI/UX 안정화
- 플로우/라우팅 정리
- 컴포넌트 정리/리팩토링
- (나중에) 백엔드 API 붙이기 용이한 구조로 정돈

기술 스택(프론트)

- React + TypeScript (Vite)
- react-router-dom
- 스타일: 현재 코드베이스 방식 유지(변경 금지)
- 상태관리: MVP는 로컬 state로 충분 (필요시 Context만)

코딩 규칙

- “동작 우선”: UI가 깨지면 안 됨
- 컴포넌트는 너무 크게 만들지 말고 pages / components 분리
- 라우팅은 App.tsx(또는 router 파일)에서 일관되게 관리
- 링크/토큰 기반 페이지(/r/:token, /helper/:token)는
  - token 유효성 검사(최소한의 형태)
  - 에러/만료 상태 UI 제공

UX 규칙(중요)

- 서비스는 WIMS 계정/비밀번호/2FA를 다루지 않는다.
- 자동 제출(봇/Selenium 등)을 전제로 UI를 만들지 않는다.
- 제출 도우미 화면은 다음 3요소를 반드시 포함한다:
  - “WIMS 열기” 버튼(새 탭)
  - “한 번에 복사” 버튼(클립보드)
  - 제출 체크리스트(단계별 체크)

파일/폴더 구조 권장

- src/pages/* 라우트 단위 화면
- src/components/* 재사용 컴포넌트
- src/lib/* 유틸(클립보드, 포맷터 등)
- src/mock/* 더미 데이터(필요 시)

변경 방식(중요)

- 한 번에 대규모 리팩토링 금지
- 변경은 PR 단위로 작게 쪼갠다
- UI 변경 시:
  - 라우팅 깨짐 여부 확인
  - 주요 화면 3개 이상 직접 확인(메인/양도등록/양수링크)

PR 규칙(필수)

- 기능 단위로 작은 PR
- 예: “메인 탭 UI 개선”, “양도 등록 폼 개선”, “제출 도우미 복사 기능 추가”
- PR 하나에 너무 많은 변경 금지
- PR 템플릿(설명에 포함):
  - 변경 요약
  - 테스트 방법(어떤 URL에서 무엇 확인했는지)
  - UI 변경이면 스크린샷 2~4장
- CodeRabbit 사용 방식
  - PR 생성 시 CodeRabbit 리뷰 요청
  - 리뷰 반영 후 머지
- PR 머지 기준:
  - 빌드 깨짐 없음
  - 라우팅 깨짐 없음
  - 핵심 플로우 최소 1회 수동 확인
